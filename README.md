## Profile

I'm a software engineer working across the frontend and backend of production e-commerce systems — the kind that carry real inventory, real checkout flows, and real users. My work spans **scalable frontend architecture, authentication, product catalogs, cart and checkout logic, admin tooling, REST API design, and deployment pipelines**, most of it on React/Next.js, Node.js/Express, and MongoDB.

I was one of two engineers selected from a large applicant pool at **AIEnterprise Inc.**, placed on a live production project during probation, and later recognized as the first engineer on my team — across all seniority levels — to receive a Spot Award. That project (BrandsMart USA) involved end-to-end ownership of the **Account and Checkout modules**: 800+ user stories and 300+ defect resolutions across a codebase built for real commercial traffic.

Outside client work, I build complete systems on my own time — not demos. Real-time chat infrastructure, cloud-backed media platforms, and internal tooling, each taken from a rough idea through to a deployed, working product.

## Engineering Philosophy

<table>
<tr>
<td width="50%" valign="top">

**Ship the boring, correct version first**
Auth, validation, and error states are not an afterthought — they're the first thing built, not the last thing patched in before a demo.

</td>
<td width="50%" valign="top">

**Architecture should survive scope creep**
Client/server separation, typed contracts, and clear module boundaries mean a project doesn't need a rewrite when requirements change.

</td>
</tr>
<tr>
<td width="50%" valign="top">

**Infrastructure choices are engineering decisions**
Object storage, CDN delivery, and deployment targets are picked deliberately — for cost, latency, and failure behavior — not by default.

</td>
<td width="50%" valign="top">

**Refactors are part of the job, not a detour**
Migrating a Replit monorepo to a proper client/server split, or converting a JS codebase to TypeScript mid-project, is routine maintenance — not a special event.

</td>
</tr>
</table>

## AWS & Cloud Architecture

**Something**'s media pipeline is the clearest example of deliberate infrastructure design in my personal project work: uploads don't touch the application server's disk at all — they stream straight to **Amazon S3**.

```txt
Client (React)
   │  multipart/form-data
   ▼
Express + Multer  ──────────────►  multer-s3  ──────────────►  Amazon S3
   │  auth / validation                (@aws-sdk/client-s3)        │
   │                                                                │
   ▼                                                                ▼
MongoDB  ◄──────────────────────  stores S3 object key       uploads/{username}/{postId}/{type}/{uuid}.ext
  (metadata only)
```

**Why S3, and not disk or MongoDB storage:**

- **Stateless application servers** — media never lands on the Express host, so the server can be redeployed or scaled without any file-sync problem.
- **Type-aware key routing** — an upload middleware inspects the file extension and routes objects into `images/`, `videos/`, `audio/`, or `documents/` prefixes within a per-user, per-post path, keeping the bucket organized at scale.
- **AWS SDK v3 (`@aws-sdk/client-s3`, `@aws-sdk/lib-storage`)** — the modular v3 client keeps bundle size down and supports managed multipart uploads for larger video/audio files instead of buffering them in memory.
- **Private ACLs by default** — objects are written as private, with access mediated through the application rather than public bucket URLs.
- **Separate upload paths for signup vs. profile updates** — profile-photo uploads resolve a deterministic key (`uploads/{username}/profile/profile-photo.jpg`) so re-uploads overwrite in place, while post media gets a UUID-based key per upload — two different S3 key strategies for two different consistency requirements.
